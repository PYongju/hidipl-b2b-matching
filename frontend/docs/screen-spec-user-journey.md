# 견적 검토 코파일럿 스크린 스펙 및 유저저니 초안

작성일: 2026-05-28  
목적: 견적 검토 대시보드의 화면 구성, 사용자 흐름, 데이터/AI 분석 구조를 팀 공통 기준으로 정리한다.

## 1. 제품 방향

견적 검토 코파일럿은 고객 요구사항과 여러 공급사 견적서를 비교하여 구매 담당자가 빠르게 검토, 보정, 최종 선정할 수 있도록 돕는 웹 기반 업무 도구이다.

최종 산출물은 앱으로 확장 가능한 반응형 웹으로 정의한다.

- 1순위: PC/노트북
- 2순위: 태블릿
- 3순위: 모바일 조회

비교표와 공급사 판단 정보가 많으므로 모바일 퍼스트가 아니라 데스크톱 퍼스트 반응형 웹 대시보드로 설계한다.

## 2. 전체 유저저니

### 2.1 기본 흐름

1. 로그인
2. 프로젝트 목록 확인
3. 새 프로젝트 생성
4. 요구사항 입력
5. 견적서 업로드
6. AI 분석 중
7. 비교 대시보드 확인
8. 비교표 수정 및 검토 메모 작성
9. AI 재분석
10. 최종 선정 확정
11. 고객 보고서 생성/내보내기

### 2.2 화면 단위

실제 화면은 모든 단계를 각각 별도 페이지로 나누기보다 아래 4개 화면으로 구성한다.

1. 프로젝트 목록 페이지
2. 새 프로젝트 생성 Wizard
3. AI 분석 중 화면
4. 비교 대시보드

현재 구현된 화면은 4번 비교 대시보드에 해당한다.

## 3. 화면별 스펙

## 3.1 프로젝트 목록 페이지

### 목적

로그인 후 사용자가 이전 프로젝트를 확인하거나 새 프로젝트를 생성하는 시작 화면이다.

### 주요 기능

- 프로젝트 목록 보기
- 진행 상태별 필터
- 새 프로젝트 생성
- 최근 수정일 기준 정렬
- 프로젝트 상세 진입

### 프로젝트 카드 정보

- 프로젝트명
- 고객사/회사명
- 설치 위치
- 현재 단계
- 비교 마감일
- 공급사 수
- 최종 추천 공급사
- 최근 수정일
- 담당자

### 상태 예시

- 작성 중
- 견적 업로드 중
- AI 분석 중
- 비교 검토 중
- 최종 선정 완료
- 보고서 생성 완료

### 반응형

- PC: 3열 카드 또는 테이블+카드 혼합
- 태블릿: 2열 카드
- 모바일: 1열 카드

## 3.2 새 프로젝트 생성 Wizard

### 목적

프로젝트 기본 정보, 비교 기준, 견적서를 입력받아 AI 분석을 시작한다.

### 추천 구성

Drawer로 시작할 수 있으나, 기능이 늘어나면 별도 페이지 또는 넓은 Wizard 화면으로 확장한다.

### Step 1. 프로젝트 기본 정보

실무에서 매칭 가부를 가르는 핵심 필드이다.

- 회사명
- 설치 위치
- 프로젝트 일정
- 활용 용도 및 디스플레이 요구사항
- 현재 단계
- 비교 마감일

비교 마감일은 공급사 데이터가 아니라 프로젝트 설정값이다. 대시보드 상단의 D-day, 정시 진행 여부, 프로젝트 목록 상태 계산에 사용한다.

### Step 2. 비교 기준 설정

- 발주처 유형
- 검토 프리셋
- 고객 우선순위
- 동일 카테고리 비교 여부
- 비요청 옵션 제외 여부
- 결측치 처리 정책

검토 프리셋 예시:

- 균형 검토
- 가격 우선
- 신뢰도 우선
- 유지보수 우선
- 스펙 우선

### Step 3. 견적서 업로드

- 공급사별 견적서 업로드
- 업로드 상태 표시
- OCR/추출 상태 표시
- 견적서 교체 업로드
- 파일 보기

### Step 4. AI 비교 시작

AI 분석 시작 전 현재 설정을 요약해서 보여준다.

- 비교 기준
- 업로드된 공급사 수
- 결측치 처리 정책
- 비요청 옵션 처리 방식

## 3.3 AI 분석 중 화면

### 목적

AI가 실제로 무엇을 처리하고 있는지 사용자에게 보여주는 전환 화면이다.

### 진행 단계 예시

1. 견적서 파일 확인
2. 금액/납기/보증 항목 추출
3. 동일 카테고리 여부 확인
4. Size/Spec 차이 확인
5. 비요청 옵션 분리
6. 결측치 정책 적용
7. 공급사별 장단점 생성
8. AI 근거 요약 생성
9. 비교 대시보드 구성

### UI 방향

단순 스피너보다 단계별 진행 상태를 보여준다.

예:

- 완료: 파일 업로드 확인
- 완료: 금액 항목 추출
- 진행 중: 비요청 옵션 분리
- 대기: AI 근거 요약 생성

## 3.4 비교 대시보드

### 목적

공급사별 견적과 AI 분석 결과를 한 화면에서 검토하고 최종 선정까지 진행하는 메인 업무 화면이다.

### 주요 영역

1. 프로젝트 상단 정보
2. 공급사 매칭 현황
3. 견적 비교 테이블
4. AI 근거 요약
5. 공급사별 장단점
6. 검토 메모
7. 최종 선정
8. 보고서/내보내기
9. 활동/히스토리

## 4.1 상단 프로젝트 정보

표시 정보:

- 프로젝트명
- 고객사/회사명
- 설치 위치
- 프로젝트 일정
- 활용 용도
- 현재 단계
- 비교 마감일
- 검토 프리셋

액션:

- 검토 건 생성
- AI 재분석
- 고객 보고서 미리보기
- 검토 진행/최종 확정

## 4.2 공급사 매칭 현황

공급사 카드에 표시할 정보:

- 공급사명
- 우선 검토/최저가/프리미엄 파트너 뱃지
- 설치 사례 수
- 성사율 상태
- 응답 속도
- 제출 상태
- 검토 상태
- 견적서 보기
- 견적서 교체
- 재분석

## 4.3 견적 비교 테이블

### 필수 노출 정보

아래 항목은 접히지 않고 항상 보여야 한다.

- 견적금액 VAT 별도
- 업력
- 매출액 (3년 평균)
- 프로젝트 수 (3년 평균)
- 회사 위치

### 하위 상세 항목

아래 항목은 아코디언으로 접고 펼칠 수 있게 한다.

- 디스플레이 H/W
- 시스템 장비
- 설치비

### 추가 비교 정보

제품명이 다르거나 스펙이 다른 경우 아래 정보를 명시한다.

- 카테고리
- 제품명
- Size
- Spec
- 동일 Spec 여부
- Spec 차이
- Size 차이

### 동일 카테고리 비교 원칙

비교 대상은 동일 카테고리끼리만 비교한다.

예:

- LED vs LED: 비교 가능
- LED P1.8 vs LED P2.0: 비교 가능하되 Spec 차이 표시
- LED vs 비디오월: 같은 비교표에서 비교 금지
- LED vs 빔프로젝터: 같은 비교표에서 비교 금지

서로 다른 카테고리는 별도 참고 견적으로 분리한다.

### 비요청 옵션 처리

고객이나 하이디플이 요청하지 않은 옵션을 일부 업체만 포함한 경우, 동일 기준 비교를 위해 별도 처리한다.

예:

- 원본 견적 총액
- 비교 기준 금액
- 비요청 옵션 항목
- 별도 코멘트

처리 원칙:

- 요청하지 않은 옵션은 총액 비교에서 제외 가능
- 단, 고객 보고서에는 별도 코멘트로 표시
- 결측치가 아니라 비요청 항목으로 분류

## 4.4 결측치 처리 정책

### 납기 결측

- 처리: 무시
- 이유: 납기는 고객 측 의사결정, 계약, 발주 일정에 연동되므로 견적 단계에서 결측이어도 큰 의미가 없다.
- UI 표시: 미기재 정도로만 표시
- AI 추천 영향: 낮게 반영

### 단가 결측

- 처리: 재요청
- 추정 금지
- 이유: 단가는 파트너사의 고유 권한 영역이므로 하이디플이 임의 추정하지 않는다.
- UI 표시: 재요청 필요, 최종 선정 전 확인 필요
- AI 추천 영향: 높게 반영

### 파트너 자체 추가 옵션

- 처리: 합산 시 별도 코멘트
- 이유: 결측이 아니라 요청하지 않은 항목이 추가된 경우이다.
- UI 표시: 비요청 옵션, 비교 제외 검토, 별도 코멘트

## 4.5 AI 근거 요약

### 목적

공급사별 비교 결과를 담당자가 빠르게 이해할 수 있도록 요약한다.

### 포함 내용

- 추천 공급사
- 추천 이유
- 가격/납기/스펙/설치/보증 관점 요약
- 확인 필요 항목
- 주의할 리스크

### 데이터 출처

AI 근거 요약은 견적서 원본만 보는 것이 아니라 아래 데이터를 함께 사용한다.

- 원본 견적서 추출값
- 사용자가 수정한 비교표 값
- 사용자가 추가한 항목
- 비교 제외 처리한 항목
- 검토 메모
- 비교 정책

## 4.6 공급사별 장단점

### 목적

각 공급사의 선택 근거와 리스크를 명확히 보여준다.

### 표시 예시

- A Display
  - 장점: 프리미엄 파트너, 설치 사례 풍부, 보증 명확
  - 단점: 총액 최저는 아님

- BrightSign Korea
  - 장점: 총액 기준 경쟁력 우수
  - 단점: 프리미엄 파트너는 아님

- VisionTech
  - 장점: 일부 항목 가격 경쟁력
  - 단점: Spec 차이, 설치 사례 기준 미달, 확인 필요 항목 존재

## 5. 수정 및 AI 재분석 구조

## 5.1 비교표 수정

견적 비교 테이블은 사용자가 수정할 수 있어야 한다.

가능한 기능:

- 행 추가
- 행 삭제
- 행 이름 수정
- 공급사별 값 수정
- 상태 변경
- 비교 제외 처리
- 수정 사유 입력

상태 예시:

- 정상
- 확인 필요
- 미기재
- 비교 제외
- 재요청 필요
- 비요청 옵션

## 5.2 AI 재분석

AI 재분석은 현재 비교표 기준으로 다시 추천하도록 설계한다.

중요 원칙:

- AI가 테이블 값을 마음대로 덮어쓰지 않는다.
- 사용자가 수정한 비교표 값을 유지한다.
- AI는 현재 기준 데이터를 입력으로 받아 요약, 장단점, 추천 결과를 다시 생성한다.

재분석 입력:

- 원본 견적서 추출값
- 사용자 수정값
- 추가 행
- 비교 제외 항목
- 검토 메모
- 비교 정책

재분석 후 갱신되는 영역:

- AI 근거 요약
- 공급사별 장단점
- 최종 추천 업체
- 확인 필요 항목
- 공급사 카드 태그/상태

## 6. 히스토리 구조

히스토리는 세 종류로 나눈다.

## 6.1 프로젝트 목록 히스토리

로그인 후 첫 화면에서 보는 이전 프로젝트 목록이다.

목적:

- 이전 견적 검토 건 찾기
- 진행 상태 파악
- 프로젝트 맥락 복원

프로젝트가 월 4~10건 발생하면 6개월 후 50건 이상 쌓이므로 전체 로그 방식보다 프로젝트 단위 그룹핑이 필요하다.

## 6.2 프로젝트 내부 활동 기록

특정 프로젝트 안에서 일어난 이벤트 기록이다.

예:

- 프로젝트 생성
- 견적서 업로드
- AI 분석 완료
- 비교표 값 수정
- AI 재분석 실행
- 최종 선정 확정
- 보고서 생성

## 6.3 AI 분석 버전 히스토리

AI 재분석이 여러 번 실행될 경우, 분석 결과를 버전 단위로 저장한다.

예:

- v1 최초 분석
- v2 설치비 확인 반영
- v3 비요청 옵션 제외 후 재분석

초기 MVP에서는 복잡한 버전 비교보다 재분석 기록만 남긴다.

추후 필요 시 사용자가 재분석 전 버전 이름을 저장할 수 있게 한다.

## 7. LLM/AI 처리 구조

UI는 LLM이 내부에서 몇 번 호출됐는지 알 필요가 없다. UI는 정해진 JSON 구조를 받아 화면에 표시한다.

추천 구조:

1. Extract 단계
   - 견적서에서 금액, 납기, 보증, 설치비 등 추출

2. Normalize/Compare 단계
   - 동일 카테고리 비교
   - 비요청 옵션 분리
   - 결측치 정책 적용
   - 비교 기준 금액 산출

3. Explain 단계
   - AI 근거 요약
   - 공급사별 장단점
   - 추천 사유
   - 확인 필요 항목 생성

## 9. MVP 우선순위

### 1차

- 프로젝트 목록 페이지
- 새 프로젝트 생성 Wizard
- AI 분석 중 화면
- 현재 비교 대시보드 연결

### 2차

- 비교표 셀 수정
- 행 추가/삭제
- 비요청 옵션 표시
- 결측치 상태 표시
- AI 재분석

### 3차

- 프로젝트 내부 활동 기록
- AI 재분석 기록
- 보고서 버전 관리
- 고객 보고서 내보내기 고도화
